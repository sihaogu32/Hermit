hermit

以 NousResearch/hermes-agent 為底座的個人化 AI agent。不是「另一個更會聊天的 AI」，而是以繁體中文與台灣生活脈絡為核心的「個人知識與任務中樞」——能持續記住偏好、跨來源整合個人資料、把答案變成可執行的提醒／草稿，且讓使用者看得見依據、控制得了記憶與權限。

✦ · 功能介紹：hermit 能幫我做什麼

這一節寫給第一次認識 hermit 的人，不需要懂程式。簡單說：hermit 不是「另一個更會聊天的 AI」，而是一位會記得你、會幫你把事情接著做完、而且你隨時看得到它用了什麼、能隨時喊停的個人助理。它說繁體中文、懂台灣的生活脈絡（日期、地名、語氣），把「查到答案」一路接到「幫你設好提醒、寫好草稿」，而每一步都讓你做主。

它能幫我做什麼

🧠

記得你說過的事

你提過的偏好、習慣、家人、常用資訊，它會記住，下次不用重講。而且這些記憶你看得到、改得動、刪得掉。

例：你說過「我喝美式不加糖、爸媽住台中」，之後它就記得。
🔗

把你的個人資料串起來（要你授權）

在你授權的範圍內，把行事曆、信箱、雲端檔案、筆記接進來，於是它能回答跟「你」有關的問題，而不只是聊公開知識。第一個接的是 Google 行事曆。

例：「我這週有哪些行程要先準備？」
📎

回答會附上出處

問它資訊型的問題，它會把答案連同「依據是哪裡來的」一起給你，方便你自己核對，而不是憑空講一段要你照單全收。

例：查一個資訊，附上來源連結讓你點開確認。
✅

把答案變成行動

不只回答，還能幫你設提醒、排定行程、寫好郵件或訊息的草稿——把「知道」接到「做完」。寄出、送出這種最後一步，仍由你親自按下。

例：「下週一提醒我繳費」、「幫我擬一封回覆信的草稿」。
🇹🇼

講你的語言、懂在地脈絡

預設繁體中文，日期、地名、語氣都照台灣習慣走，不是翻譯腔。

例：講「下週三」「下個月初」它懂，地名口語也對得上。
🔒

你完全掌控

它要把任何事記進你的檔案、或替你對外做事，都會先問過你；高風險的事第一版根本不做。詳見下方「你永遠是按下確認的人」。

例：它想記住一件事，會先列出來等你勾選確認。

它跟一般 AI 聊天哪裡不一樣

一般 AI 聊天hermit
這次講完，下次多半就忘了持續記得你的偏好，而且記憶你能看、能改、能刪
只能聊它訓練過的公開知識在你授權下，接你的行事曆／信箱／檔案，回答跟「你」有關的事
給你一段話，出處難查研究型回答附上依據來源，方便你核對
通常停在「給你答案」把答案接成提醒、排程、草稿等實際行動
它在背後做了什麼，你不一定看得到任何寫入你的資料或對外動作，都要你先確認

你永遠是按下確認的人

安心，是這個產品的底線，不是加分項

  • 不會偷偷自動動：要把任何事記進你的個人檔案、或替你對外做事（例如寄信），它都會先「提出建議」，等你親自確認才會真的發生。
  • 高風險的事第一版不做：付款、下單、刪除這類動作，第一版刻意不碰。
  • 記憶完全可控：它記住的東西，你隨時看得到、改得動、刪得掉。
  • 資料在你自己的地盤：跑在你自己的環境裡，狀態集中保存、可備份。

適合誰、現在做到哪

適合誰：想要一個「真的記得我、幫我把事情收尾、又讓我安心掌控」的個人助理。產品建議從 26–35 歲、已習慣使用 AI 的知識工作者起步。

現在做到哪（白話版）：基礎骨架已搭好，目前正在做最關鍵也最難的一塊——把第一個個人資料源（Google 行事曆）安全接進來，並建立「先徵得你同意才寫入」的同意機制。語音輸入、更多資料源、行動 App 等會分階段推出。

本節為產品設計目標與分階段規劃；部分功能仍在開發或規劃中，實際以各階段釋出為準。想看技術細節與目前進度，請往下看第 1–9 節。

1 · 專案定位與市場前提

hermit 是以 NousResearch/hermes-agent 為底座的個人化 AI agent，定位在三交集：繁中／台灣在地化 × 權限可控的個人資料脈絡 × 從研究到提醒到草稿的閉環執行。與法務專案 hermes_law 平行獨立、共用底座與模式但不搬法務資產，唯一關鍵新工為「個人資料 connector + 權限同意中心」。

一句話定位（seed-spec §0 / README）

  • 以 NousResearch/hermes-agent 為基底的個人化 AI agent。
  • 不是「另一個更會聊天的 AI」，而是以繁體中文與台灣生活脈絡為核心的「個人知識與任務中樞」。
  • 四項能力：持續記住偏好、跨來源整合個人資料、把答案變成可執行的提醒／草稿、讓使用者看得見依據且控制得了記憶與權限。
  • 差異化窗口在三交集：繁中／台灣在地化 × 權限可控的個人資料脈絡 × 從研究到提醒到草稿的閉環執行。
  • 正面比模型能力是紅海；可打的是「在繁中環境下更可信地處理個人任務」。

與 hermes_law 的關係（seed-spec §1）

  • 專案邊界：獨立專案、獨立 repo，與 hermes_law（法務）平行，不是它底下的 vertical。
  • 共用：同一個 hermes-agent 上游底座 + hermes_law 已驗證的架構模式。
  • 不共用：不共用 running instance；不搬法務資產（legal_kb、citation-guard、fsc-penalty-search、MOJ pipeline 全留在 hermes_law）。
  • Runtime：seed-spec §1 原寫「獨立 OS user，沿用上游預設 ~/.hermes 與 ~/wiki，靠 $HOME 與法務完全隔離」。
  • 為什麼獨立而非 vertical：runtime 狀態、設計紅線、產品 identity 全切乾淨——個人版不必背法務 citation 紅線與 KB 維運，法務也不被個人資料 connector 的隱私複雜度污染。代價是各自維護一套 HERMES_HOME 與備份鏡像。

市場前提（seed-spec §2，公開資料基礎版）

  • 台灣 GenAI 使用率 43.19% / 付費訂閱率 8.54% → 使用面已主流、付費仍早期，Freemium 起手最合理。
  • 台灣曾用 46% / 自費 17% / 每日重度 9% → 已有習慣化族群但自掏腰包仍少數。
  • 最佳起始族群：26–35 歲（自費率最高約 23%），「高頻＋付費＋留存」飛輪最可能成形。
  • 信任風險感知：個資外洩自評影響 36.6%；過半民眾對 AI 日常化更擔心 → 信任是採用門檻、不是加分項。
  • 個人 Plus 約 NT$149–249/月（低於國際旗艦 US$20、高於零售工具 App）；Pro 約 NT$299–499；B2B2C 因企業／機構代付仍高於個人自費而為成長主軸。

缺口分析（seed-spec §4，最重要的一節）

  • 報告的 MVP 幾乎全部落在現成底座上，唯一關鍵新工是「個人資料 connector + 權限同意中心」。
  • 直接用（無缺口）：可控長期記憶 / 提醒排程 / 多 persona / skill 擴充 / 多平台 / 繁中（hermes core + profile + gateway + SOUL.md 繁中 native）。
  • 小缺口：回覆可引用 / 來源透明 → citation-guard 四層 hook 架構直接用、換來源。
  • 中缺口：隱私「不靜默自動改／動作」 → legal-kb-admin human-confirm pattern 直接用、改成 connector 同意中心。
  • critical path（最大新工）：跨來源個人脈絡（行事曆／郵件／雲端檔／筆記）——只有單一 legal KB 來源，無現成。
  • 中缺口（可繞過／後置）：行動端原生前台（先 messaging gateway 起步）、語音輸入／相機理解。
  • 低缺口（可後置）：裝置端 / hybrid 路由 / 離線（報告自列次要）。
  • 結論：先把 connector + 同意中心做出來（真個人化的心臟），其餘走現成底座，hybrid／離線／原生 app 後置。

MVP 範圍（seed-spec §6）

MVP 只需證明三件事：(1) 它真的比通用聊天更懂我；(2) 它能把答案變成行動；(3) 它讓我感到可控而非冒犯。

首版必做（現成底座）

  • 繁中與台灣語境優化（SOUL.md 繁中 native）、對話核心 + 短長期記憶。

首版必做（需新工）

  • 行事曆／提醒／待辦（cron 現成、connector 需新工）。
  • 雲端檔案／筆記摘要 1–2 個高價值來源（connector + 同意中心需新工）。
  • 引用式研究回答（來源透明 guard 需移植）；語音輸入（gateway + STT，中等新工）。
  • 郵件／訊息草稿：生成現成，郵件 connector 視範圍。

首版明確不做（高風險）

  • 第三方服務操作（支付／訂票／下單）。
  • 高風險領域（醫療／投資／法律）結論代理——首版只做資訊整理、不做結論代理。
  • MVP 後段／後置：裝置端離線摘要／搜尋。

目標族群與商業模式

  • 起始族群：建議鎖 26–35 歲（自費率最高約 23%）的知識工作者，是「高頻＋付費＋留存」飛輪最可能成形的族群（seed-spec §2、§10 開放決策#5 待拍板）。
  • 商業模式：Freemium + 訂閱 + B2B2C 綁定；免費層導流，個人 Plus NT$149–249/月，Pro NT$299–499 解鎖代理工作流。
  • B2B2C（電信／金融／教育／裝置）是成長主軸而非可選項——報告指企業／機構代付目前仍高於個人自費。

法規與合規（seed-spec §9）

  • 個資法：蒐集告知、特定目的與合法基礎、安全措施、外洩通報 → 做成產品功能（權限畫面、刪除／匯出、敏感欄位遮罩、審計日誌、事故回報）。
  • AI 基本法（2026 起）：七原則（永續／人類自主／隱私／資安／透明可解釋／公平／問責）+ AI 產出適當揭露標記 → 高風險回答強制 disclaimer、產出可標記。
  • 首版避開醫療／投資／法律結論代理、支付／訂票／自動下單等高風險場景。
  • 中國法域不視為自然延伸（另成體系）；目標市場聚焦繁中／台灣。

核心關鍵概念

cloud-first + 預留 hybrid 插槽：報告主張 Hybrid（on-device + cloud + 權限中心 + 記憶治理）為理想，但承認 cloud-first 可接受起步。hermit 起步走 cloud-first，預留 hybrid 路由插槽（on-device 能力後置）。

2 · 5 條設計紅線

設計紅線 single source = docs/seed-spec.md §8，改寫自 hermes_law 4 條 + 新增 1 條。紅線#3「不動 core」要求所有客製落在四個擴充點；紅線#1 約束 connector 與 skill 不得 hardcode 情境；紅線#2 由來源透明 guard 落實；紅線#5 強制寫入／對外路徑走人工確認的同意中心；紅線#4 把整個 runtime 包進隔離環境（Docker / HERMES_HOME）。

紅線 #1 · 個人情境／領域是 first-class

內涵：不把單一情境 hardcode 進 prompt/skill/tool 名；按情境分層（取代 hermes_law 的 jurisdiction first-class）。

為什麼：讓新增 health/finance/family 變成「加一個情境 + 一組 skill + 一個 connector」可橫向擴充。

如何落地：約束 connector（Slot 1）與 reusable skill（Slot 3）的命名與分層。

紅線 #2 · 來源透明／可引用是必要防線

內涵：研究型回答附引用、個人化回答可追溯觸發來源；走 hook 強制，不只 prompt 紀律（移植 citation-guard）。

為什麼：信任是採用門檻，紀律不可靠、需 hook 強制。

如何落地：來源透明 guard（Slot 4，P2）＋ 來源驗證工具。

紅線 #3 · 不動 hermes-agent core 就不動

內涵：客製先走 skill / profile / config / plugin / tool；動 core 是最後手段（走 fork）。

為什麼：保留上游可升級性，把客製集中在擴充點。

如何落地：所有客製落在四個 Extension Slot；core module（agent/gateway/hermes_cli/cron）不動。

紅線 #4 · 執行環境隔離

內涵：seed-spec §8 原文為「per-project 隔離（採獨立 OS user）」，沿用上游預設 ~/.hermes 與 ~/wiki、靠 $HOME 隔離；README/CLAUDE.md 已更新為「走 Docker，HERMES_HOME 在容器內 /root/.hermes」。

為什麼：狀態集中、可攜、鏡像備份。

如何落地：整個 runtime 包進 Docker / HERMES_HOME 隔離環境。

紅線 #5 · 個人資料「不靜默自動動作」（新增）

內涵：任何寫入個人資料或對外執行的路徑都要有明確同意／人工確認入口；高風險動作（支付／下單／刪除）首版不自動（移植 legal-kb-admin human-confirm）。

為什麼：對應隱私信任門檻與個資法／AI 基本法。

如何落地：connector（Slot 1）的寫入／對外路徑強制走 machine-proposes / human-confirms 的權限同意中心。

3 · 架構與目錄責任

hermit 的真實 runtime（容器內 /root/.hermes，含巢狀 hermes-agent source 與 ~/wiki）不入 git；repo 只保存三個性質不同的 git 鏡像。整個 HERMES_HOME 不入 git，避免 runtime 狀態（state.db、sessions/、logs/、auth.json）污染 git。

HERMES_HOME 模型

  • HERMES_HOME：hermes runtime 的根目錄。Docker 內固定為 /root/.hermes（上游預設）；本機 host 預設 ~/.hermes（本機實況 /home/laura/.hermes）。
  • 解析規則：HERMES_HOME=${HERMES_HOME:-$HOME/.hermes}。
  • 巢狀 hermes-agent source：NousResearch/hermes-agent 整棵 source tree 巢狀在 /root/.hermes/hermes-agent。整棵是 upstream、不入 git；只有我方擴充（新增檔／patch）鏡像到 patches/hermes-agent/ 受 git 追蹤。

state / code / knowledge 各歸其位對照表（核心約定#3）

類型位置
可執行程式／測試真實在 ~/.hermes/hermes-agent/；repo 只存鏡像 patches/hermes-agent/files/（修改檔走 diffs/）
hermes 個人知識庫（llm-wiki）wiki/ — 鏡像 runtime ~/wiki（hermes /llm-wiki 產物、使用者個人知識；非 repo 開發紀錄處）
runtime 狀態~/.hermes/sessions/、~/.hermes/logs/、~/.hermes/state.db* 等（不入 git、不入 overlay）
設計／安裝筆記docs/（seed-spec / install-runtime / roadmap / port-sources）
開發擴充的紀錄歸 Claude（CLAUDE.md / 記憶 / docs/），不寫進 wiki/（核心約定#4）

客製化擴充點（不動 hermes-agent core，核心約定#2 / 紅線#3）

  • /root/.hermes/SOUL.md：agent 整體人格、語氣、邊界（繁中 native、來源透明、不靜默自動動作）。
  • /root/.hermes/config.yaml：模型、tool、gateway、UI 設定（不含 secrets）。
  • /root/.hermes/plugins/<name>/：dashboard plugin（P1：connector + 同意中心）。
  • /root/.hermes/skills/<domain>/<skill>/：自製 skill。

白名單機制與排除項

  • overlay：刻意不用 skills/* plugins/* blanket glob，避免掃進上游約 90 個 bundled skill、且踩到 stock skill 內的 secret placeholder。
  • wiki 排除項：manifest.sh、.gitkeep（mirror-only 檔，不可被 --delete 清掉也不回灌 runtime）、personal/cache、__pycache__、*.pyc、.DS_Store、*.swp、*.tmp。
  • Secrets 兜底掃描：export 與 import 前都跑 run_secrets_audit，掃 overlay 與 wiki 鏡像內 *.yaml/*.yml/*.md/*.json/*.env/*.toml 的 api_key/secret/password/token 等欄位；行尾為空字串（''/""）視為安全，其他非空命中即 abort。

export 流程（實況 → 鏡像，commit 前用）

  • 前置硬檢查：HERMES_HOME 與 $HERMES_HOME/hermes-agent 必須存在，否則 die。
  • EXPORT overlay：依 HERMES_OVERLAY_PATHS，目錄用 rsync -a --delete（排除 __pycache__/*.pyc/*.pyo/.DS_Store/*.swp/*.tmp），單檔用 rsync -a。

  • 前置硬檢查：.hermes-overlay/ 與 $HA_DIR 必須存在，否則 die；先跑一次 run_secrets_audit。

  • IMPORT diffs：對每個 .patch 先 git apply --check 預演；可套就 apply，已套過（reverse --check 乾淨）則跳過，否則 warn 並標記失敗，最後若有失敗則 die。
  • docker build 期自動跑 import 把 overlay+patches 疊回容器內 runtime。
