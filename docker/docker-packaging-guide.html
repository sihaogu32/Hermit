1打包模型 A / B / C / D

repo 只存「調校的差異」(A)；core (B)、機密 (C)、狀態 (D) 各自處理。

A你的調校 · 烤進 image

  • .hermes-overlay/（SOUL / config / 3 plugin / fsc-penalty skill / memories）
  • patches/hermes-agent/（legal_kb、verify_citation、pipeline、moj 下載 + tests）
  • wiki/legal/knowledge_base/（62 部法規 KB，2.3MB）
✓ COPY 進來 + sync_overlays.sh import

B上游 core · clone 鎖版本

  • NousResearch/hermes-agent，巢狀在 $HERMES_HOME/hermes-agent
  • venv + 195 packages + 74 bundled skills
✓ git clone + checkout CalVer tag

CSecrets · 不打包

  • ~/.hermes/.env（LLM 憑證）、~/.hermes/auth.json
✗ runtime 注入（-e / -v）

DRuntime 狀態 · 不打包

  • ~/.hermes/{sessions,logs,state.db,legal_kb_scans}、wiki/legal/sources(MOJ raw)
✗ volume 或 ephemeral
C 不打包 ≠ 不能填 image 仍保留 runtime 注入口：使用者 -e OPENAI_API_KEY=... 或 -v ...:/root/.hermes/.env 帶自己的憑證。

2巢狀版面 + base／A 兩段式

image 內忠實重現修正後的真實版面：HERMES_HOME 在家目錄，source 巢狀其下。

/root/.hermes/                 # = HERMES_HOME（容器內 ~ 即 /root）
├── hermes-agent/              # B：巢狀的 core（鎖版本 clone + setup）
│   └── venv/                  # 195 packages
├── plugins/ skills/ memories/ # A：sync_overlays import 還原進來
├── SOUL.md  config.yaml
└── .env                       # C：runtime 掛載，不在 image 內

/opt/hinagiku/                 # A：repo 鏡像（COPY 進來）
└── wiki/legal/knowledge_base/ # KB；HERMES_LEGAL_KB_DIR 指這裡
① clone hermes @CalVer tag→ setup-hermes.sh（venv+deps） ══ base 完成 ══ ② COPY repo (A)→ sync_overlays.sh import→ py_compile 健檢

① base＝乾淨、鎖死版本的 hermes（可單獨做成 hermes-base:<tag> 重用）；② 疊 A＝把你的調校 import 上去。sync_overlays.sh import 此時把 overlay 寫進 /root/.hermes、patches 寫進 /root/.hermes/hermes-agent（腳本已改吃 HERMES_HOME）。

3版本鎖定：別用「0.14.0」鎖

0.14.0 不是唯一的 git ref 0.14.0 是 pyproject.toml 的 package version，它橫跨多個 release、不唯一;上游真正的 release 是 CalVer git tag（如 v2026.5.16、v2026.5.7…）。所以：
  • 要可重現 → --build-arg HERMES_AGENT_REF=<SHA 或 tag>；建議用完整 commit SHA（唯一、不受 tag 有無影響）。
  • 版本敏感度：patches/diffs/（改上游既有檔的 patch）的 git apply 才會挑版本——ref 與 export 當下不符就會在 ② 失敗。本專案目前 diffs/ 為空（擴充全是 files/ 新增檔），這種硬失敗暫時不會發生;但新檔會呼叫上游 API,版本差太遠仍可能在 runtime 出錯,故仍建議釘在你開發/測試的版本。
  • image 內 /root/.hermes/.hermes-agent-pinned-sha 會記下實際 checkout 的 commit 供追溯。
hermes-agent 是 git editable 安裝（非 PyPI wheel），所以鎖 git tag/commit、不是 pip install ==0.14.0。

查你已安裝的 hermes-agent 版本（填進 HERMES_AGENT_REF）

# 精確 commit SHA（最保險，建議直接用這個）
git -C ~/.hermes/hermes-agent rev-parse HEAD

# 想要 CalVer tag：HEAD 正好落在 tag 上才有輸出（淺 clone 可能為空，需先 git fetch --tags）
git -C ~/.hermes/hermes-agent tag --points-at HEAD

4Dockerfile 逐段

完整檔在隨附的 Dockerfile；關鍵段落：

ARG HERMES_AGENT_REF=v2026.5.16            # B：鎖 CalVer tag（非 0.14.0）
ENV HERMES_HOME=/root/.hermes              # 巢狀版面的 HERMES_HOME
RUN apt-get install -y git ripgrep rsync && pip install uv

# ① base：鎖版本的 hermes，巢狀在 $HERMES_HOME/hermes-agent
RUN git clone "$HERMES_AGENT_REPO" "$HERMES_HOME/hermes-agent" \
 && git -C "$HERMES_HOME/hermes-agent" checkout "$HERMES_AGENT_REF"
RUN cd "$HERMES_HOME/hermes-agent" && yes n | env HERMES_HOME="$HERMES_HOME" bash setup-hermes.sh

# ② 疊上 A：repo 鏡像 → sync import 到 $HERMES_HOME
COPY . /opt/hinagiku/
RUN cd /opt/hinagiku && bash scripts/sync_overlays.sh import
ENV HERMES_LEGAL_KB_DIR=/opt/hinagiku/wiki/legal/knowledge_base   # KB 在 repo 內

EXPOSE 9119 8642
ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]
CMD ["web"]

base image 用 nikolaik/python-nodejs:python3.11-nodejs20（Python 3.11 對齊 setup 的 3.11.15；Node 20 給 web 前端）。

5.dockerignore 是安全關鍵

別把本地 ~/.hermes 烤進 image Docker build context 不看 .gitignore。若你在「已跑起來」的環境 build,本地的 hermes-agent/（venv 數 GB）與 .hermes/（含 secrets、state.db）可能被 COPY . 烤進 image。隨附 .dockerignore 明確排除它們,強制走「clone 鎖版本 + sync import」重建：

hermes-agent/   .hermes/        # 防止 C/D 外洩
wiki/legal/sources/   wiki/legal/logs/   # L1 raw / changelog，數 GB
.git/  .claude/  **/__pycache__/
（wiki/legal/knowledge_base/ 不排除——那是 A，要烤進去。）

6Build & Run

① 檔案位置（docker/ + repo 根的 .dockerignore）

docker/Dockerfile、docker/entrypoint.sh、本指南都在 docker/；.dockerignore 則必須在 hinagiku/ 根——Docker 只認 build context 根的 .dockerignore。build 一律從 repo 根執行（context = repo 根）。

② Build（鎖 CalVer tag）

docker build -f docker/Dockerfile --build-arg HERMES_AGENT_REF=v2026.5.16 -t hermes-legal:latest .

③ Run（C：secrets runtime 注入）

# 管理後台 + 法規 plugins（9119）
docker run --rm -p 9119:9119 \
  -v $PWD/secrets/hermes.env:/root/.hermes/.env:ro \
  hermes-legal:latest

# 或 chat gateway（8642）
docker run --rm -p 8642:8642 -e OPENAI_API_KEY=$OPENAI_API_KEY hermes-legal:latest run

# 或互動 CLI
docker run --rm -it -v $PWD/secrets/hermes.env:/root/.hermes/.env:ro hermes-legal:latest cli

④（選用）持久化 runtime 狀態 D

docker run -p 9119:9119 -v $PWD/secrets/hermes.env:/root/.hermes/.env:ro \
  -v hermes_sessions:/root/.hermes/sessions \
  -v hermes_logs:/root/.hermes/logs \
  hermes-legal:latest

7對外暴露（loopback 的坑）

預設 bind 127.0.0.1 hermes web(9119) 與 gateway(8642) 預設綁 127.0.0.1、CORS 只放行 localhost。同機 -p 9119:9119 開 http://localhost:9119 通常 OK;要從別台/區網連,容器內服務綁 loopback、-p 透不出去,需讓 hermes bind 0.0.0.0（依版本設定）或在前面架 reverse proxy。詳見你 repo 的 docs/custom-frontend-architecture.md。
